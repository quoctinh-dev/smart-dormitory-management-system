# FACE-BACKEND-01: Face Domain Implementation Blueprint

## 1. Executive Summary
This document serves as the implementation blueprint for the Face Domain within the SDMS Backend. It is derived entirely from an audit of the `docs/07-face/` technical architecture specifications. This blueprint dictates the exact entities, events, API contracts, and security configurations required to transform the Face module from an empty shell into a robust, decoupled Spring Boot bounded context.

*(Note: No source code is generated in this phase. This is the official blueprint).*

## 2. Face Aggregate Matrix (Domain Model)
The Face domain strictly separates physical identity from biometric metadata to protect privacy and decouple AI integrations.

| Entity | Type | Ownership | Key Attributes | Lifecycle Responsibility |
| --- | --- | --- | --- | --- |
| `FaceProfile` | Aggregate Root | Face Module | `profileId`, `studentId`, `faceImageUrl`, `status`, `rejectionReason` | Manages the approval workflow for student photos. |
| `FaceEmbedding` | Entity | Face Module | `embeddingId`, `profileId`, `vector` (512d float array), `modelVersion` | Manages the raw biometric vector data generated by the AI engine. |

**Required Enum:**
- `FaceApprovalStatus`: `PENDING_APPROVAL`, `APPROVED`, `REJECTED`

## 3. Face Database Matrix
To fix the `V15` migration leakage, biometric tracking must be extracted from the `students` table.

| Table Name | Primary Key | Foreign Keys | Specific Tech Requirements | Migration Strategy |
| --- | --- | --- | --- | --- |
| `face_profiles` | `profile_id` (UUID) | `student_id` (UNIQUE) | | Extract `face_image_url` from `students` into this table. |
| `face_embeddings` | `embedding_id` (UUID) | `profile_id` (CASCADE) | Requires PostgreSQL `pgvector` extension | Create table explicitly to store 512-dimension vectors. |

## 4. Face Event Matrix
The Face Module must communicate asynchronously with the `Student`, `Notification`, and `Smart Access` modules to prevent cyclic dependencies.

| Event | Publisher | Consumer(s) | Trigger Condition |
| --- | --- | --- | --- |
| `FacePhotoUploadedEvent` | Student Controller | Face Service | Student submits a photo via Mobile App. |
| `FaceProfileApprovedEvent` | Face Service | Student Service, Notification | Admin approves photo, triggering AI extraction. |
| `FaceProfileRejectedEvent` | Face Service | Notification Service | Admin rejects photo; student must re-upload. |
| `FaceEmbeddingExtractedEvent` | Face Service | Smart Access Service | 512d vector generated; ready for KTX gates. |

## 5. Face API Matrix
The REST API layer is split between Student-facing endpoints and Admin review dashboards.

### Student APIs
| Method | Endpoint | Request Payload | Response Payload | Description |
| --- | --- | --- | --- | --- |
| `POST` | `/api/v1/student/face/register` | `FaceRegistrationRequest` | `FaceProfileResponse` | Submits initial portrait photo url. |
| `POST` | `/api/v1/student/face/re-register` | `FaceRegistrationRequest` | `FaceProfileResponse` | Submits new photo after a rejection. |
| `GET` | `/api/v1/student/face/me` | None | `FaceProfileResponse` | Retrieves current profile status. |

### Admin APIs
| Method | Endpoint | Request Payload | Response Payload | Description |
| --- | --- | --- | --- | --- |
| `POST` | `/api/v1/admin/face/profiles/{profileId}/approve` | None | `Void` | Approves photo, triggering AI extraction. |
| `POST` | `/api/v1/admin/face/profiles/{profileId}/reject` | `FaceRejectionRequest` | `Void` | Rejects photo, providing a reason string. |

## 6. Face Security Matrix (Permission Catalog)
To align with the `smartaccess` standard, hardcoded string roles are banned. The following constant registry must be created:

| Permission Constant | Mapped Role(s) | Applied To |
| --- | --- | --- |
| `FacePermissions.FACE_REGISTER` | `ROLE_STUDENT` | `/student/face/register`, `/re-register` |
| `FacePermissions.FACE_VIEW_SELF` | `ROLE_STUDENT` | `/student/face/me` |
| `FacePermissions.FACE_APPROVE` | `ROLE_ADMIN`, `ROLE_STAFF` | `/admin/face/profiles/{id}/approve` |
| `FacePermissions.FACE_REJECT` | `ROLE_ADMIN`, `ROLE_STAFF` | `/admin/face/profiles/{id}/reject` |

## 7. AI Service Integration Matrix
The SDMS Spring Boot backend delegates mathematical extraction to a stateless FastAPI microservice.
- **Contract:** `POST /api/v1/ai/extract`
- **Request:** `{ "imageUrl": "https://..." }`
- **Response:** `{ "success": true, "processingTimeMs": 142, "embedding": [0.0124, ...] }`
- **Failure Handling:** If the AI Service is down or fails to detect a face, the `FaceProfile` remains `APPROVED` but the `FaceEmbedding` generation is queued for retry, or flagged for manual admin intervention.

## 8. Implementation Order
1. **Database:** Write Flyway migration `V22__extract_face_domain.sql` to create `face_profiles` and `face_embeddings` (with `pgvector`).
2. **Domain:** Implement `FaceProfile` and `FaceEmbedding` JPA Entities and Enums.
3. **Security:** Create `FacePermissions` constant registry.
4. **Integration:** Build `FaceAiClient` (Feign or RestTemplate) to contact FastAPI.
5. **Business Logic:** Implement `FaceProfileService` and publish standard Application Events.
6. **API:** Expose Controllers mapped with `FacePermissions`.
